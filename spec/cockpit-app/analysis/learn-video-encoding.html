第 01 节

一帧是什么（What is a Frame）

视频就是快速播放的一连串静止图片。每张图片就是一帧——一个由彩色像素组成的网格。

1920 x 1080 = 2,073,600 像素
x 3 字节（R, G, B）= ~6 MB 每帧
x 30 fps = 180 MB/s
x 60 秒 = 10.8 GB 每分钟

第 02 节

I 帧、P 帧、B 帧——帧类型（Frame Types）

不是每一帧都需要存完整画面。编码器利用时间冗余——帧与帧之间大部分画面是不变的。

I-Frame（关键帧）

完整画面。可以独立解码。类似一张 JPEG 快照。

P-Frame（预测帧）

只存储与前一帧的差异。比 I-frame 小约 70%。

B-Frame（双向预测帧）

同时参考前后帧。体积最小。比 I-frame 小约 85%。

GOP——图片组（Group of Pictures）

帧按重复模式组织成 GOP。一个典型的模式：

I-Frame（关键帧） P-Frame（预测帧） B-Frame（双向预测帧）
NextFrame 把 HTML 录制为截图——每一帧都是 I-frame。编码器（VideoToolbox / x264）必须分析它们并创建 P/B 帧来压缩。
第 03 节

编解码器 vs 容器（Codec vs Container）

Codec = 语言

如何压缩和解压视频数据。把原始像素转换成紧凑比特流的算法。

例：H.264, H.265, VP9, AV1

Container = 信封

如何把视频 + 音频 + 元数据打包到一个文件里。只是一个封装格式。

例：.mp4, .webm, .mov, .mkv

编解码器 容器 画质 速度 兼容性
H.264 .mp4 良好 快 通用
H.265 / HEVC .mp4 更好 较慢 大多数
VP9 .webm 良好 慢 网页端
AV1 .mp4 / .webm 最佳 非常慢 仅新设备
ProRes .mov 近无损 快 Apple/专业
NextFrame 使用 H.264 + .mp4——在所有平台和播放器上兼容性最好。

第 05 节

码率控制（Bitrate Control）

三种方式告诉编码器花多少比特：

CBR

固定码率——全程码率不变。适合直播，但在简单画面上浪费比特。

VBR

可变码率——复杂画面多用比特，简单画面少用。每字节画质更好。

CRF

质量目标——编码器自动决定码率。最适合离线渲染。NextFrame 使用此模式。

码率随时间变化

CBR 在简单画面上浪费比特（水平线一直很高）。VBR 和 CRF 在复杂度高的地方分配更多比特——中间的尖峰就是复杂画面。

CRF 让编码器智能分配码率。对于离线渲染（非直播），它能产出最高的性价比画质。

注意事项

音频必须单独处理。你不能像拆分帧渲染那样拆分连续音频流。

为什么帧纯净（Frame-Pure）很重要

每个 WebView 可以独立跳到任意帧。React/基于状态的渲染器做不到——它们需要顺序更新状态。

并行录制之所以可行，是因为 NextFrame 的时间线是帧纯净（frame-pure）的——任意帧都可以独立渲染，不需要知道其他帧的状态。
第 09 节

音视频同步（Audio-Video Sync）

音频是连续波形，视频是离散帧。保持同步需要精确的时间戳。

核心问题

视频
音频
PTS
0.0s0.5s1.0s1.5s2.0s2.5s3.0s

PTS（Presentation Timestamp，展示时间戳）

每一帧和每个音频样本都有时间戳："在第 3.000 秒展示我"。如果 30fps 的第 90 帧应在 3.0 秒展示但在 3.03 秒才出现——你就会看到唇音不同步。

NextFrame 的做法

TTS 生成带有词级时间戳的音频。视频帧根据同样的时间戳计算。同一个数据源 = 同步有保障。

音视频通过共享时间戳实现同步。NextFrame 从同一组 TTS 词级时间戳推导音频和帧的时序——同步是结构性的，不是事后校正。
